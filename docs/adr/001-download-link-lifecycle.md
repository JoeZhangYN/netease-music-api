# ADR-001: 下载链接生命周期管理

## 背景

网易云音乐 API 返回的下载 URL 是临时的、有时效的，部分链接可能是一次性的（首次 HTTP 请求后失效）。在重构过程中多次遇到以下问题：

1. 开发或调试时对下载 URL 发起 HEAD 请求"验证有效性"，导致链接被消耗，后续真正下载失败
2. 下载失败后用同一个已过期 URL 反复重试，浪费时间且永远不会成功
3. 缓存了下载 URL 供多次使用，但 URL 在时效窗口后失效
4. 在日志中输出完整 URL，包含鉴权 token，存在安全风险

## 决策

### 将所有下载 URL 视为一次性链接

即使部分 URL 可能是可重用的，也统一按最严格的一次性语义处理。这样做的代价（偶尔多一次 API 调用）远小于收益（消除整类 bug）。

### 明确唯一消耗点

`download_file_ranged()` 是系统中唯一允许向下载 URL 发起 HTTP 请求的函数。其他任何地方都禁止对下载 URL 发起网络请求。

### URL 获取与下载原子化

`get_music_info()` 获取 URL 后，应尽快传递给 `download_file_ranged()` 执行下载。中间不做额外的网络操作、不做长时间等待。

### 失败后重新获取

下载最终失败时，丢弃旧 URL，从 `get_song_url()` 重新获取。不在外层用旧 URL 重试。

## 被否决的方案

### 方案 A: URL 有效性预检（HEAD 请求）

在下载前发 HEAD 请求检查 URL 是否有效。

- 否决原因：部分 CDN 将 HEAD 视为一次消耗，预检本身会导致链接失效
- 否决原因：增加一次网络往返，却无法保证 GET 时仍然有效（TOCTOU 问题）

### 方案 B: URL 缓存与复用

缓存下载 URL，同一首歌的多次下载复用同一个 URL。

- 否决原因：URL 有时效窗口，缓存的 URL 必然过期
- 否决原因：一次性链接被第一次下载消耗后，缓存值无效
- 否决原因：增加了缓存失效逻辑的复杂度，收益不确定

### 方案 C: URL 封装为智能指针（引用计数）

用 `Arc<OnceCell<String>>` 封装 URL，控制消耗次数。

- 否决原因：过度工程化，当前场景 URL 只在单一流程中使用
- 否决原因：增加了抽象层但没有解决根本问题（网络操作的幂等性）

## 后果

### 正面

- 消除了一整类"链接被意外消耗"的 bug
- 简化了心智模型：URL 从获取到消耗是单向线性流程
- 下载失败的恢复策略清晰：重新获取 URL 即可

### 负面

- 每次下载重试都需要额外一次 API 调用获取新 URL（成本可接受）
- 不能利用"某些 URL 实际上可重用"这一特性做优化

### 约束

1. `download_file_ranged()` 内部重试（5 次）是对同一 URL 的网络层重试，这是合理的
2. 外层重试（如用户重新点击下载）必须重新获取 URL
3. 批量下载中，每首歌独立获取各自的 URL，不共享
